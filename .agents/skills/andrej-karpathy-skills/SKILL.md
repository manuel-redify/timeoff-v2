# Andrej Karpathy Skills for Opencode

## Overview

This skill provides behavioral guidelines derived from Andrej Karpathy's observations on LLM coding pitfalls. It helps reduce common mistakes by promoting four key principles: Think Before Coding, Simplicity First, Surgical Changes, and Goal-Driven Execution.

## When to Use This Skill

- Use when you want to improve the quality and reliability of code generated by LLMs
- Use when you notice LLMs making assumptions without checking
- Use when you want to prevent overcomplication and bloated code
- Use when you want to ensure changes are surgical and traceable to user requests
- Use when you want to transform tasks into verifiable goals with clear success criteria

## How It Works

### Principle 1: Think Before Coding

**Don't assume. Don't hide confusion. Surface tradeoffs.**

Before implementing:
- State assumptions explicitly. If uncertain, ask.
- If multiple interpretations exist, present them - don't pick silently.
- If a simpler approach exists, say so. Push back when warranted.
- If something is unclear, stop. Name what's confusing. Ask.

### Principle 2: Simplicity First

**Minimum code that solves the problem. Nothing speculative.**

- No features beyond what was asked.
- No abstractions for single-use code.
- No "flexibility" or "configurability" that wasn't requested.
- No error handling for impossible scenarios.
- If you write 200 lines and it could be 50, rewrite it.

Ask yourself: "Would a senior engineer say this is overcomplicated?" If yes, simplify.

### Principle 3: Surgical Changes

**Touch only what you must. Clean up only your own mess.**

When editing existing code:
- Don't "improve" adjacent code, comments, or formatting.
- Don't refactor things that aren't broken.
- Match existing style, even if you'd do it differently.
- If you notice unrelated dead code, mention it - don't delete it.

When your changes create orphans:
- Remove imports/variables/functions that YOUR changes made unused.
- Don't remove pre-existing dead code unless asked.

The test: Every changed line should trace directly to the user's request.

### Principle 4: Goal-Driven Execution

**Define success criteria. Loop until verified.**

Transform tasks into verifiable goals:
- "Add validation" → "Write tests for invalid inputs, then make them pass"
- "Fix the bug" → "Write a test that reproduces it, then make it pass"
- "Refactor X" → "Ensure tests pass before and after"

For multi-step tasks, state a brief plan:
```
1. [Step] → verify: [check]
2. [Step] → verify: [check]
3. [Step] → verify: [check]
```

Strong success criteria let you loop independently. Weak criteria ("make it work") require constant clarification.

## Examples

### Example 1: Implementing a Feature

Instead of jumping straight to implementation:
1. First, state assumptions about the requirements
2. Consider alternative interpretations if ambiguous
3. Propose the simplest solution that could work
4. Define clear success criteria (e.g., "Write tests for edge cases, then implement until they pass")

### Example 2: Refactoring Code

Instead of making broad "improvements":
1. Identify exactly what needs to change based on user request
2. Make only those specific changes
3. Ensure any removed code was made unused by your changes
4. Match existing code style even if you prefer different approaches
5. Verify tests pass before and after

## Best Practices

### ✅ Do This

- State assumptions explicitly before coding
- Ask for clarification when uncertain
- Propose simpler alternatives when they exist
- Write only the minimum code needed to solve the problem
- Make surgical changes that touch only what's necessary
- Clean up only code that your changes made unused
- Match existing code style and conventions
- Define clear, verifiable success criteria
- Break multi-step tasks into verifiable steps
- Loop until goals are achieved

### ❌ Don't Do This

- Don't make assumptions without stating them
- Don't hide confusion or uncertainty
- Don't pick silently when ambiguity exists
- Don't add features beyond what was requested
- Don't create abstractions for single-use cases
- Don't add speculative flexibility or configurability
- Don't refactor code that isn't broken
- Don't "improve" adjacent code or comments unnecessarily
- Don't delete pre-existing dead code unless asked
- Don't make changes that can't be traced to user requests
- Don't work with vague success criteria like "make it work"

## Customization

These guidelines are designed to be used with project-specific instructions. They work best when combined with specific technical guidelines for your project.

For project-specific rules, consider adding sections like:
- Use TypeScript strict mode
- All API endpoints must have tests
- Follow existing error handling patterns in src/utils/errors.ts
- Specific architectural patterns or conventions

## Tradeoff Note

These guidelines bias toward caution over speed. For trivial tasks (simple typo fixes, obvious one-liners), use judgment — not every change needs the full rigor.

The goal is reducing costly mistakes on non-trivial work, not slowing down simple tasks.
